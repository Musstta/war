# v0.40 Audit Resolutions

This document records which findings from `docs/systems-audit-v0.39.md` were addressed by the v0.40 Economic & Prestige Rebalance.

## Addressed in v0.40

### ¬ß3.1 ‚Äî Prestige as a one-way ratchet (no meaningful decay)
**Audit finding:** Prestige accumulated from territories, treaties, and war wins but had no decay sources ‚Äî a dominant nation staying passive would never lose Prestige.

**Resolution:** `computePrestige` now subtracts four decay sources:
- `warsLost √ó PRESTIGE_LOSS_PER_WAR_DEFEAT (10)` ‚Äî cumulative, incremented in tick.ts on peace deal defeat
- `territoriesLost √ó PRESTIGE_LOSS_PER_TERRITORY_LOST (8)` ‚Äî cumulative, incremented on cession/conquest
- `revoltTerritoryCount √ó PRESTIGE_DECAY_PER_REVOLT_TICK (1)` ‚Äî per-tick per revolting territory
- `PRESTIGE_DECAY_PER_INSOLVENCY_TICK (2)` ‚Äî per tick while wealthStock < 0

New DB fields: `Nation.warsLost Int @default(0)`, `Nation.territoriesLost Int @default(0)`.
Migration: `20260611020000_prestige_decay`.

### ¬ß3.2a ‚Äî Trade route loss with no Prestige/wealth consequence
**Audit finding:** `applyRouteLossEvent` applied unrest via TerritoryModifier but had no effect on Prestige or nation wealth ‚Äî a severed grown route was costless beyond local unrest.

**Resolution:** `applyRouteLossEvent` now accepts `nations` parameter and applies:
- `prestigePenalty = round(lossFraction √ó PRESTIGE_LOSS_PER_ROUTE_LOSS)` to owner (half to partner)
- `wealthPenalty = lossFraction √ó SHIPMENT_LOSS_WEALTH_VALUE` to destination nation

Both scale with `lossFraction = (currentCapacity - baseCapacity) / growthCap`, so un-grown routes still incur no penalty.

### ¬ß3.2c ‚Äî Tribute with no diminishing returns for wealthy receivers
**Audit finding:** A nation with 100+ wealth receiving 10 wealth/tick tribute was treated identically to a nation with 0 wealth ‚Äî no incentive dynamics, tribute was always worth receiving at face value.

**Resolution:** Effective tribute now scales: `effectiveTribute = amount √ó (1 ‚àí min(CAP=0.5, receiverWealth / SCALE=100))`. Payer still loses full amount; diminished portion is lost (models marginal utility, not a transfer back). Additionally: if `payerTerritories / receiverTerritories < EXPLOITATION_THRESHOLD=0.4`, receiver incurs TerritoryModifier unrest + Prestige penalty per tick.

### ¬ß3.3a ‚Äî No infrastructure maintenance cost
**Audit finding:** Ports, markets, and roads generated benefits (trade capacity, unrest reduction, movement speed) with no ongoing cost ‚Äî building them was always strictly positive.

**Resolution:** Per-tick maintenance deducted from nation wealth in the army-upkeep loop:
- Port: `portLevel √ó PORT_MAINTENANCE_PER_LEVEL (0.1/level/tick)`
- Market: `MARKET_MAINTENANCE_FLAT (0.05/tick)`
- Road: `ROAD_MAINTENANCE_FLAT (0.02/tick)`

Fort maintenance excluded ‚Äî deferred to v0.41 with garrison rework.

### ¬ß3.4 ‚Äî Population transfer compat-scaling bug (`void shockMagnitude`)
**Audit finding:** The compat-scaled shock magnitude was correctly computed (`shockMagnitude = (1 - compatScore) √ó POPULATION_TRANSFER_UNREST_SCALE`) but immediately discarded via `void shockMagnitude`. All territories received the same fixed `POPULATION_TRANSFER_UNREST_SCALE` regardless of compat.

**Resolution:** New `TerritoryState.populationTransferShockMagnitude: Float @default(0)` field stores the per-territory compat-scaled value. The territory loop uses this stored value instead of the fixed constant. Additionally fixed a `nationCultures` forward-reference runtime error at the transfer site by computing receiver culture inline.
